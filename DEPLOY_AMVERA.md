# Amvera deployment and rollback runbook

Этот документ не содержит значений секретов и не разрешает автоматический
деплой. Все production-действия выполняются только после отдельного
подтверждения.

## Зафиксированная исходная точка

- GitHub baseline: `00beb1bfd2c83c743aad4c9961e435c945de7760`.
- Подтверждённый внутренний commit Амверы: `bb8cec43f8268a21f314ce4677c00e2e108fab54`.
- Публичный endpoint: `https://suvvy-tourvisor.premium-world.ru/tour-search`.
- Контейнер слушает порт `80`.
- `amvera.yml` не используется.

Перед деплоем нужно повторно убедиться, что внутренний commit Амверы содержит
тот же код, что и зафиксированный GitHub baseline.

## Production gates

Деплой запрещён, пока не выполнены все условия:

1. Утверждён финальный список `active_contract`.
2. `config/operator_registry.json` содержит минимум один активный ID.
3. Версия реестра не равна `UNCONFIRMED`.
4. Локальные тесты и CI прошли.
5. Зафиксирован точный Git commit новой версии.
6. Сохранены названия текущих переменных Амверы без вывода значений.
7. Подготовлен rollback на предыдущую версию.

## Переменные новой версии

Добавить или проверить названия:

```text
APP_ENVIRONMENT=production
SERVICE_VERSION=0.4.0
GIT_COMMIT_SHA=<exact commit>
OPERATOR_REGISTRY_PATH=config/operator_registry.json
ENABLE_DEBUG_ENDPOINTS=false
EXPOSE_API_DOCS=false
SUVVY_ALLOW_BODY_TOKEN=true
```

Существующие секретные переменные не копировать в тикеты, логи или чат.
`MOCK_TOURVISOR` в production должен оставаться `false`.

## Порядок развёртывания

1. Синхронизировать в Амверу только утверждённый Git commit.
2. Запустить сборку Docker-образа.
3. Убедиться, что образ содержит директории `app/` и `config/`.
4. Запустить новую реплику.
5. Проверить `GET /health` — без обращения к Tourvisor.
6. Проверить `GET /ready`:
   - `status=ok`;
   - ожидаемая версия приложения;
   - ожидаемая версия/hash whitelist;
   - ненулевой `allowed_operator_count`.
7. Не выполнять поиск до отдельного разрешения на живой API-тест.
8. После разрешения выполнить один заранее согласованный запрос.
9. Проверить наличие `operatorIds` и отсутствие результатов вне whitelist.
10. После серверной проверки отдельно переключить SUVVY на Bearer и новые поля ответа.

## Rollback

1. Остановить переключение SUVVY, если оно ещё не завершено.
2. Вернуть предыдущий рабочий commit и совместимую конфигурацию.
3. Не менять домен и DNS.
4. Проверить `/health`.
5. Если откат вернул версию без enforced whitelist, отключить автоматическую
   выдачу и передавать обращения менеджеру. Не продолжать unrestricted-поиск.
6. Зафиксировать `request_id`, версию приложения и whitelist, на которых
   возникла ошибка, без копирования секретов.
